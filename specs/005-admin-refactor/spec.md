## User Scenarios & Testing *(mandatory)*

### User Story 1 - 관리자 전용 헤더를 통한 페이지 내비게이션 (Priority: P1)

관리자가 로그인 후 관리자 전용 헤더를 통해 제조사, 판매차량, 출고차량 관리 페이지 간에 자유롭게 이동할 수 있다. 메인 사이트의 헤더와 푸터는 관리자 페이지에서 표시되지 않으며, 관리자 전용 헤더만 노출된다.

**Why this priority**: 관리자 페이지의 독립적인 내비게이션 구조는 모든 후속 기능(페이지 분리, 목록/등록/수정 페이지)의 기반이 되므로 가장 높은 우선순위를 가진다.

**Independent Test**: 관리자 로그인 후 전용 헤더가 표시되고, 각 메뉴 버튼 클릭 시 해당 관리 페이지로 이동되는지 확인하여 독립적으로 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리자가 로그인한 상태, **When** 관리자 페이지에 진입, **Then** 메인 사이트의 헤더와 푸터는 보이지 않고 관리자 전용 헤더만 표시된다
2. **Given** 관리자 전용 헤더가 표시된 상태, **When** 헤더 좌측의 로고를 클릭, **Then** 메인 사이트 홈페이지로 이동한다
3. **Given** 관리자 전용 헤더가 표시된 상태, **When** "제조사" 버튼을 클릭, **Then** 제조사 관리 페이지로 이동한다
4. **Given** 관리자 전용 헤더가 표시된 상태, **When** "판매차량" 버튼을 클릭, **Then** 판매차량 관리 페이지로 이동한다
5. **Given** 관리자 전용 헤더가 표시된 상태, **When** "출고차량" 버튼을 클릭, **Then** 출고차량 관리 페이지로 이동한다
6. **Given** 관리자 전용 헤더가 표시된 상태, **When** "로그아웃" 버튼을 클릭, **Then** 로그아웃 처리 후 로그인 페이지로 이동한다
7. **Given** 현재 페이지에 해당하는 헤더 메뉴 버튼, **When** 관리자가 해당 페이지에 있을 때, **Then** 현재 페이지에 해당하는 버튼이 활성화 상태(하이라이트)로 표시된다

---

### User Story 2 - 관리 대상별 별도 페이지 분리 (Priority: P1)

현재 하나의 관리자 페이지에서 탭으로 전환하는 구조를 제조사, 판매차량, 출고차량 각각 별도의 페이지(URL)로 분리한다. 각 관리 대상은 목록 페이지, 등록 페이지, 수정 페이지를 독립적으로 가진다.

**Why this priority**: 페이지 분리는 관리자 경험의 핵심 구조 변경이며, 목록/등록/수정 기능의 전제 조건이므로 P1이다.

**Independent Test**: 각 관리 대상(제조사/판매차량/출고차량)의 목록 페이지에 직접 URL로 접근하여 해당 데이터가 올바르게 표시되는지 확인하여 독립 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리자가 로그인한 상태, **When** 제조사 관리 페이지 URL에 접근, **Then** 제조사 목록이 표시된다
2. **Given** 관리자가 로그인한 상태, **When** 판매차량 관리 페이지 URL에 접근, **Then** 판매차량 목록이 표시된다
3. **Given** 관리자가 로그인한 상태, **When** 출고차량 관리 페이지 URL에 접근, **Then** 출고차량 목록이 표시된다
4. **Given** 각 관리 대상의 목록 페이지, **When** "등록" 버튼을 클릭, **Then** 해당 관리 대상의 등록 페이지로 이동한다
5. **Given** 각 관리 대상의 목록 페이지, **When** 특정 항목의 행을 클릭, **Then** 해당 항목의 상세 페이지로 이동한다

---

### User Story 3 - 목록 페이지 페이지네이션 및 정보 표시 (Priority: P2)

각 관리 대상의 목록 페이지에서 데이터를 테이블 형태로 표시하되, 각 행에는 제목, 작성자 닉네임, 작성날짜('2025-01-23' 형식)가 표시되며, 목록 하단에 페이지네이션이 배치된다.

**Why this priority**: 목록 페이지의 사용성 개선이며, 페이지 구조가 먼저 분리된 이후에 적용할 수 있으므로 P2이다.

**Independent Test**: 목록 페이지에서 다수의 데이터가 페이지네이션으로 분할 표시되는지, 각 행의 정보가 올바른 형식으로 보이는지 확인하여 독립 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리 대상의 목록 페이지, **When** 데이터가 존재할 때, **Then** 각 행에 제목, 작성자 닉네임, 작성날짜('YYYY-MM-DD' 형식)가 표시된다
2. **Given** 목록 데이터가 한 페이지 분량을 초과할 때, **When** 목록 페이지를 조회, **Then** 목록 하단에 페이지네이션 컨트롤이 표시된다
3. **Given** 페이지네이션이 표시된 상태, **When** 다음 페이지 버튼을 클릭, **Then** 다음 페이지의 데이터가 표시된다
4. **Given** 목록 데이터가 없을 때, **When** 목록 페이지를 조회, **Then** "데이터가 없습니다" 등의 빈 상태 안내가 표시된다

---

### User Story 4 - 상세 페이지에서 수정 및 삭제 (Priority: P2)

각 관리 대상의 상세 페이지에서 해당 데이터의 정보를 확인하고, 수정 버튼과 삭제 버튼을 통해 데이터를 관리할 수 있다.

**Why this priority**: 데이터 관리의 핵심 기능이지만 목록 페이지와 페이지 분리가 선행되어야 하므로 P2이다.

**Independent Test**: 특정 데이터의 상세 페이지에서 정보 확인 후 수정/삭제 기능이 동작하는지 확인하여 독립 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리 대상의 상세 페이지, **When** 페이지에 진입, **Then** 해당 데이터의 모든 정보가 표시된다
2. **Given** 상세 페이지에서 수정 버튼이 보이는 상태, **When** "수정" 버튼을 클릭, **Then** 수정 페이지로 이동한다
3. **Given** 상세 페이지에서 삭제 버튼이 보이는 상태, **When** "삭제" 버튼을 클릭, **Then** 삭제 확인 모달이 표시된다
4. **Given** 삭제 확인 모달이 표시된 상태, **When** 삭제를 확인, **Then** 해당 데이터가 삭제되고 목록 페이지로 이동한다

---

### User Story 5 - 메탈 계열 테마 색상 적용 (Priority: P3)

관리자 페이지 전체에 블루 계열이 아닌 메탈 계열 색상을 적용한다. globals.css에 정의된 primary, secondary 색상을 최대한 활용하여 일관된 디자인을 구현한다.

**Why this priority**: 시각적 개선이므로 기능적 요구사항이 완료된 후 적용할 수 있어 P3이다.

**Independent Test**: 관리자 페이지 전체를 시각적으로 확인하여 블루 계열 색상이 제거되고 메탈 계열 색상이 적용되었는지 확인하여 독립 테스트 가능하다.

**Acceptance Scenarios**:

1. **Given** 관리자 페이지에 진입한 상태, **When** 페이지를 조회, **Then** 모든 UI 요소가 메탈 계열 색상(primary, secondary 기반)으로 표시된다
2. **Given** 관리자 페이지의 버튼, 헤더, 테이블 등, **When** 색상을 확인, **Then** 블루 계열(accent 색상)이 아닌 메탈 계열(steel gray, silver 등)이 사용된다
3. **Given** globals.css에 정의된 색상 변수, **When** 관리자 페이지 스타일링에 적용, **Then** primary(#4a5568), secondary(#a0aec0) 계열 색상이 주로 사용된다

---

### Edge Cases

- 관리자가 로그인하지 않은 상태에서 관리자 페이지 URL에 직접 접근하면 로그인 페이지로 리다이렉트된다
- 존재하지 않는 관리 대상 ID로 상세/수정 페이지에 접근하면 적절한 오류 안내가 표시된다
- 목록 데이터가 0건일 때 페이지네이션은 표시되지 않고 빈 상태 안내만 표시된다
- 삭제 확인 모달에서 취소를 클릭하면 데이터가 삭제되지 않고 상세 페이지로 돌아간다
- 브라우저 뒤로가기 시 이전 관리 페이지로 정상적으로 돌아간다

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 관리자 페이지(`/admin` 하위 경로)에서 메인 사이트의 헤더와 푸터가 표시되지 않아야 한다
- **FR-002**: 관리자 전용 헤더는 좌측부터 로고(메인페이지 링크), 제조사 버튼, 판매차량 버튼, 출고차량 버튼, 로그아웃 버튼 순서로 배치되어야 한다
- **FR-003**: 관리자 헤더의 로고는 메인 사이트와 동일한 logo.png를 사용하며, 클릭 시 메인 페이지(`/`)로 이동해야 한다
- **FR-004**: 제조사 관리는 별도 페이지로 분리되어 목록, 상세, 등록, 수정 페이지를 각각 가져야 한다
- **FR-005**: 판매차량 관리는 별도 페이지로 분리되어 목록, 상세, 등록, 수정 페이지를 각각 가져야 한다
- **FR-006**: 출고차량 관리는 별도 페이지로 분리되어 목록, 상세, 등록, 수정 페이지를 각각 가져야 한다
- **FR-007**: 각 관리 대상의 목록 페이지 하단에 페이지네이션이 배치되어야 한다
- **FR-008**: 목록의 각 행에는 제목, 작성자 닉네임, 작성날짜('YYYY-MM-DD' 형식)가 표시되어야 한다
- **FR-009**: 각 관리 대상의 상세 페이지에서 해당 데이터의 정보를 조회할 수 있어야 한다
- **FR-010**: 상세 페이지에 수정 버튼과 삭제 버튼이 존재해야 한다
- **FR-011**: 관리자 페이지 전체에 메탈 계열 색상을 적용하고, globals.css의 primary/secondary 색상을 활용해야 한다
- **FR-012**: 현재 활성화된 페이지에 해당하는 헤더 메뉴 버튼은 시각적으로 구분되어야 한다

### Key Entities

- **제조사(Manufacturer)**: 자동차 제조사 정보. 코드, 이름, 카테고리(국산/수입), 로고, 정렬순서, 노출여부를 포함
- **판매차량(SaleCar)**: 판매 가능한 차량 정보. 제조사 연결, 차량명, 설명, 썸네일, 렌트/리스 가격, 뱃지, 노출여부를 포함
- **출고차량(ReleasedCar)**: 출고 완료된 차량 정보. 차량명, 썸네일, 출고일, 노출여부를 포함
- **관리자(Admin User)**: 관리 시스템에 접근 가능한 사용자. 닉네임을 포함하여 목록에서 작성자로 표시됨

## Assumptions

- 페이지네이션의 페이지당 항목 수는 10개를 기본값으로 한다
- "제목"은 각 관리 대상의 대표 명칭 필드를 의미한다 (제조사: name, 판매차량: name, 출고차량: car_name)
- "작성자 닉네임"은 데이터를 생성한 관리자의 닉네임을 의미한다
- 관리자 로그인 페이지는 기존과 동일하게 유지하되 메탈 계열 테마는 적용한다
- 상세 페이지는 목록에서 특정 행을 클릭하여 진입하는 구조이다
- 기존 API 엔드포인트는 그대로 활용하되, 페이지네이션을 위한 파라미터가 추가될 수 있다

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 관리자 페이지에서 메인 사이트 헤더/푸터가 100% 노출되지 않는다
- **SC-002**: 관리자가 헤더 내비게이션을 통해 3개 관리 영역(제조사, 판매차량, 출고차량)에 2클릭 이내로 접근할 수 있다
- **SC-003**: 각 관리 대상별로 독립된 URL을 가진 목록, 상세, 등록, 수정 페이지가 존재한다
- **SC-004**: 목록 페이지에서 페이지네이션을 통해 전체 데이터를 탐색할 수 있다
- **SC-005**: 상세 페이지에서 수정/삭제 작업을 완료하는 데 3단계 이내의 조작으로 가능하다
- **SC-006**: 관리자 페이지의 모든 UI 요소가 메탈 계열 색상으로 통일되어 있다
